# Task 16 - fetchConversations 페이지네이션 분석

## 현재 구현 분석
- 대상 함수: `src/composables/useChat.js`의 `fetchConversations()`
- 현재 쿼리: `messages` 테이블에서 `sender_id=me OR receiver_id=me` 조건으로 최신순 조회 후 `.limit(500)` 적용
- 반환 데이터: 메시지 + sender/receiver 프로필 조인
- 후처리: 클라이언트에서 `Map`으로 상대방 기준 그룹화
  - 첫 메시지를 최신 메시지로 사용
  - `!isMe && !is_read` 조건으로 conversation별 unreadCount 누적

## .limit(500) 사용 의도와 병목
### 의도
- 대화 상대가 여러 명이어도 대화 목록(최근 메시지 + 안읽은 수)을 한 번에 계산하려는 목적
- 서버 API/RPC 추가 없이 프론트 단에서 완결하려는 구현

### 병목
- **과조회**: 대화 목록 1회 로드마다 최대 500행을 가져와 JS 루프/Map 그룹화를 수행
- **네트워크 비용**: 메시지 본문/파일 메타/프로필 조인까지 포함되어 payload가 큼
- **인덱스 활용 한계**: 스키마상 인덱스는 `(sender_id, receiver_id, created_at desc)`와 `receiver_id where is_read=false` 중심이라,
  `sender_id OR receiver_id` + 정렬 쿼리는 데이터가 커질수록 비효율 가능성 존재
- **정확도 상한**: unreadCount는 최근 500건 내에서만 계산되어, 이미 완전한 전수 집계는 아님

## 옵션 평가

## Option A - Supabase RPC로 서버사이드 그룹화
### 장점
- DB에서 최신 메시지 1건/상대방 + unread 집계를 수행해 전송량 대폭 감소
- 클라이언트 그룹화 비용 제거

### 단점
- RPC SQL/권한/유지보수 추가 필요
- 현재 스프린트 범위 대비 변경 폭 큼(검증 비용 증가)
- 메시지/프로필 조인, RLS 경계, 테스트 보강까지 고려하면 리스크 상승

## Option B - 현재 방식 유지 + limit 축소
### 장점
- 코드 변경 최소, 회귀 리스크 낮음
- 즉시 체감 가능한 네트워크/클라이언트 처리량 감소
- 기존 테스트/구독 로직과 충돌 가능성 낮음

### 단점
- 근본 구조(클라이언트 그룹화)는 유지
- unreadCount 상한이 500 -> 100으로 더 낮아지는 trade-off 존재

## Option C - 가상 스크롤/무한 스크롤
### 장점
- 대량 데이터 UX 확장성 우수

### 단점
- 현재 화면 목표(대화 목록 요약)와 맞지 않음
- 상태/커서/UI 복잡도 과도 증가, 구현 대비 효과 낮음

## 선택안
- **선택: Option B**
- **이유**:
  1. 현재 목표가 "안전한 성능 개선"이므로 최소 변경으로 효과를 내는 방향이 적합
  2. Option A는 장기적으로 좋지만 이번 작업 범위를 넘어서는 DB/RPC 변경과 검증 비용이 큼
  3. Option C는 문제-해결 적합도가 낮고 복잡도 대비 실익이 작음

## 구현 결정
- `fetchConversations()`의 조회 상한을 **500 -> 100**으로 조정
- 하드코딩 값을 상수화해 의도 명시 및 추후 조정 용이성 확보
